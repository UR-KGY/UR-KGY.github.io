## 1. NetworkTransport 핵심 멤버의 내장 로직과 역할
ServerClientId { get => 0; }
역할: 유니티 Netcode 코어 엔진이 '서버(방장)'의 ID 번호를 식별하기 위한 고정 고유 라벨.

작동 원리: 내부 연산 시 엔진이 서버의 주소를 자동으로 찾아가도록 설계됨.

PollEvent
전통적인 역할: 유니티 엔진이 매 프레임 업데이트 타이밍에 트랜스포트를 직접 돌며 수신된 패킷(바이트 배열)이 있는지 확인하고 가져가는 입구.

비워둔 이유: 스팀웍스 라이브러리는 레이트 업데이트에서 패킷 수신을 진행하며, 패킷이 확인되는 즉시 콜백을 통해 유니티 엔진에 다이렉트로 데이터를 꽂아주는 이벤트 주도형 방식을 채택하고 있음. 따라서 규칙상 껍데기만 구현하고 내부는 비워둠.

Initialize
역할: 유니티 네트워크 매니저가 게임 시작 시 트랜스포트 컴포넌트를 최초 구동하는 함수. 내부에서 스팀 P2P 릴레이 서버망을 활성화함.

매개변수를 쓰지 않는 이유: 부모 클래스의 규칙상 네트워크 매니저를 인자로 받도록 강제되어 정의는 해두었으나, 스팀 트랜스포트 고유 로직상 엔진 매니저의 내부 정보가 필요하지 않아 내부에서는 무시함.

GetCurrentRtt
역할: 패킷이 상대방에게 도달했다가 돌아오는 왕복 시간(RTT, 핑)을 밀리초(ms) 단위로 측정하는 함수.

수정 및 커스텀: 라이브러리 업데이트로 인해 가벼운 통신 상태 구조체를 반환하므로, 내부의 소문자 핑 필드를 추출하여 반환하도록 커스텀함.

## 2. 데이터 순환 흐름 및 타이밍 구조
전체 사이클 (스팀 ➡️ 유니티 ➡️ 스팀)
스팀 수신 (LateUpdate): 모든 게임 오브젝트의 연산이 끝난 프레임 최하단에서 수신 함수를 실행, 랜카드에 도착한 스팀 패킷 수집.

유니티 토스 (OnMessage): 수집 즉시 유니티 엔진 코어에 데이터 전달 및 RPC/논리 연산 실행.

스팀 재발송 (Send): 유니티 엔진이 연산 결과를 다른 유저에게 공유하기 위해 트랜스포트의 송신 함수를 호출, 스팀 선로에 패킷을 다시 태워 보냄.

## 3. 네트워크 배송 옵션 변환
매핑 단순화의 이유
유니티는 5가지 배송 옵션을 세분화하여 제공하지만, 스팀웍스에는 1대1 대칭 구조가 존재하지 않음.

유니티 Netcode의 5가지 배송 옵션과 스팀웍스의 제한된 옵션 간의 실제 물리적 작동 차이는 다음과 같음.

| 유니티 `NetworkDelivery` 옵션 | 스팀웍스 `SendType` 매핑 결과 | 실제 작동 방식 및 데이터 유실/순서 제어 비하인드 |
| :--- | :--- | :--- |
| **`Unreliable`**<br>(비신뢰 우편) | **`Unreliable`** | * **동작:** 패킷 유실을 허용하며, 도착 순서도 보장하지 않음.<br>* **용도:** 데이터가 중간에 사라지거나 뒤바뀌어도 무방한 초당 수십 번의 단순 위치 동기화 등에 사용. |
| **`UnreliableSequenced`**<br>(순서 보장 비신뢰) | **`Unreliable`** (단순화) | * **유니티의 의도:** 유실은 허용하되, 최신 패킷보다 먼저 발송된 '과거 패킷'이 뒤늦게 도착하면 폐기 처리하는 방식.<br>* **스팀의 한계:** 스팀 API에는 이 옵션이 존재하지 않아 단순 `Unreliable`로 처리됨. 이로 인해 유니티가 의도한 과거 패킷 자동 폐기 기능은 작동하지 않음. |
| **`Reliable`**<br>(신뢰 등기 우편) | **`Reliable`** | * **동작:** 패킷 유실 시 재전송을 통해 100% 도착을 보장함.<br>* **특징:** 스팀의 `Reliable`은 기본적으로 내부에서 순서 정렬을 지원하므로 유니티의 의도대로 정확히 매핑됨. |
| **`ReliableSequenced`**<br>(순서 보장 신뢰 등기) | **`Reliable`** | * **유니티의 의도:** 유실 없이 100% 전달하면서, 발송된 순서 그대로 엄격하게 정렬되어 도달하는 방식.<br>* **스팀의 처리:** 스팀은 `SendType.Reliable` 하나만 지정해도 **네트워크 망 자체에서 패킷의 순서 정렬을 자동으로 수행**함. 따라서 별도의 전용 타입 없이도 유니티의 요구 사항이 완벽히 충족됨. |
| **`ReliableFragmentedSequenced`**<br>(대용량 조각화 순서 보장) | **`Reliable`** | * **유니티의 의도:** 대용량 데이터(맵 정보 등)를 전송할 때, 최대 패킷 크기 제한에 걸리지 않도록 엔진 단에서 데이터를 여러 조각으로 쪼개어 안전하게 보낸 뒤 정렬하는 방식.<br>* **스팀의 처리:** 스팀 릴레이 소켓은 `Reliable` 모드 사용 시, **내부 하위 레이어에서 대용량 패킷의 조각화와 조립을 알아서 처리**함. 개발자가 수동으로 쪼갤 필요가 없어 스팀의 기본 신뢰 타입으로 정상 작동함. |

비신뢰(Unreliable) 계열: 스팀에 순서 보장형 비신뢰 타입이 없으므로 단순 비신뢰 옵션으로 수렴.

신뢰(Reliable) 계열: 스팀은 신뢰 옵션 하나만 지정해도 내부 엔진에서 자동 순서 정렬 및 대용량 조각화를 지원하므로 모두 신뢰 매핑으로 통합 처리함.

## 4. 로비 시스템과 유니티 세대교체 흐름
역사적 배경 및 세대교체
과거 구형 (UNET 시절): 네트워크 매니저를 직접 상속받아 오버라이드하여 제어함.

현재 NGO: 네트워크 매니저를 상속하지 않는 완전한 독립 코어 컴포넌트로 사용. 규칙 제어(뇌)는 매니저가 담당하고 연결 통로(다리)는 트랜스포트가 담당하는 구조로 이원화됨. 개발자는 별도의 로비 매니저를 파서 이벤트를 구독하는 방식을 지향함.

StartClient 호출 타이밍의 진실
해당 함수는 인게임 전투 시작 신호가 아닌, 서버와 통신할 수 있는 가상의 네트워크 로그인(통신망 개설) 행위임.

따라서 스팀 로비에 입장하자마자 호출하는 것이 정석이며, 이를 통해 로비 내 채팅 및 준비 상태 UI 동기화를 진행함.

인게임 실제 진입은 통신선이 연결된 상태에서 방장이 네트워크 씬 매니저를 호출하여 모든 클라이언트를 동기화 이동시키는 방식을 사용함.

## 5. 로비와 인게임의 프리팹 스폰 제어
Default Player Prefab의 문제점
기본 플레이어 프리팹 슬롯에 인게임 전투용 캐릭터를 등록하면 로비 입장과 동시에 대기실 화면에 캐릭터가 강제 자동 생성되는 모순 발생.

해결책 (정석 커스텀 기법)
네트워크 매니저 인스펙터에서 기본 플레이어 프리팹 슬롯을 비워둠.

인게임 씬 로드가 완료된 직후, 서버(방장) 스크립트에서 각 클라이언트의 명단을 돌아가며 소유권을 명시하는 코드를 통해 수동으로 인게임 캐릭터를 스폰 제어함.

## 6. 방장 위임(Host Migration) 및 접속 제어 구조
방장 위임의 네트워크 메커니즘
방장이 퇴장할 경우 남은 클라이언트들이 기존 네트워크를 종료하고 재조립하는 과정.

기존 유니티 네트워크 매니저 및 트랜스포트 통신 셧다운.

남은 유저 중 차기 방장 선출 후, 새 방장 컴퓨터에서 호스트 가동 (새 서버 개설).

나머지 일반 참가자들은 새 방장의 스팀 ID를 추출하여 트랜스포트에 주입 후 클라이언트 재가동 및 데이터 복구 연산 진행.

호출 전제 조건
권한별 네트워크 가동 함수가 실행되기 직전 단계에서 반드시 트랜스포트 내부에 타깃 스팀 ID 주소판이 먼저 세팅되어 있어야 정상적으로 작동함.
