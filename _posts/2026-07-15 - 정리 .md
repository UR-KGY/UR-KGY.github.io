[기록] VR 멀티플레이어 기술 구현 및 아키텍처 검증
1. 네트워크 씬 로딩 시스템 (Netcode Scene Management)
Netcode Scene Loading 및 상태 관리
멀티플레이어 환경 내 클라이언트 간 동기화 및 안정적인 씬 전환을 위해 Unity Netcode 비동기 씬 로드 기능을 채택함. 반환 타입인 SceneEventProgressStatus (Enum)를 통해 로딩 파이프라인 상태를 추적하고 트러블슈팅을 진행함.

SceneEventProgressStatus.Started: 씬 전환 명령이 서버 및 클라이언트 네트워크 파이프라인에 정상적으로 전달되어 로딩이 시작됨(성공).

SceneEventProgressStatus.SceneEventInProgress: 이전 씬의 로드/언로드 연산이 완료되지 않은 상태에서 중복 씬 전환 요청 시 발생하는 예외 에러.

SceneEventProgressStatus.InvalidSceneName: 요청한 씬 이름의 철자 오류 또는 Build Settings의 씬 목록에 등록되지 않은 경우 발생하는 예외 에러.

2. 네트워크 오브젝트 생명주기 (NetworkBehaviour Lifecycle)
NetworkBehaviour 클래스는 MonoBehaviour 수명 주기 내부에서 네트워크 스폰/디스폰 연산과 긴밀히 결합되어 구동됨. 흐름 및 시점별 특징을 아래와 같이 정리함.

[Awake] ➔ [OnEnable] ➔ [OnNetworkSessionChanged] ➔ [OnNetworkSpawn] ➔ [Start]
                                                               ⬇
[OnNetworkPostSpawn] ⬳ [Update / FixedUpdate (IsOwner 체크)] ⬳ [OnNetworkDespawn] ➔ [OnDisable] ➔ [OnDestroy]
❶ 초기화 및 스폰 단계 (Initialization & Spawn)
Awake() (MonoBehaviour)

시점: 오브젝트 메모리 생성 최초 순간.

특징: 네트워크 미연결 상태이므로 권한 프로퍼티(IsOwner 등) 작동 안 됨. 순수 컴포넌트 캐싱(GetComponent) 전용으로 제한함.

OnEnable() (MonoBehaviour)

시점: 오브젝트가 활성화(activeSelf = true)될 때마다 실행.

OnNetworkSessionChanged(NetworkSessionEvent)

시점: 유니티 6 및 최신 Netcode 세션 상태 변경 시 가장 먼저 실행.

OnNetworkSpawn()  (핵심)

시점: 서버에 의해 네트워크 스폰이 공식 완료된 순간.

특징: IsOwner, IsServer, IsClient 등 네트워크 권한 체크가 100% 보장되는 시점. 로컬 카메라 활성화 및 RPC 이벤트 구독 등 권한 분기 초기화는 반드시 이 시점에 수행함.

Start() (MonoBehaviour)

시점: OnNetworkSpawn() 완료 후, 첫 번째 Update 프레임 실행 직전 1회 실행.

❷ 런타임 업데이트 단계 (Runtime & Updates)
FixedUpdate(), Update(), LateUpdate() (MonoBehaviour)

특징: 멀티플레이 시 타인 오브젝트 연산으로 인한 패킷 및 CPU 낭비를 막기 위해, 최상단에 if (!IsOwner) return; 소유권 가드를 필수 적용함.

OnNetworkPostSpawn()

시점: 씬 내 대기 중인 모든 네트워크 오브젝트들이 스폰을 완료한 바로 다음 프레임. 타 오브젝트와의 유기적 링크 작업에 활용함.

❸ 디스폰 및 소멸 단계 (Despawn & Destruction)
OnNetworkDespawn()  (핵심)

시점: 서버의 Despawn() 명령, 네트워크 세션 단절, 혹은 씬 전환으로 인해 오브젝트 소멸 직전 실행.

특징: OnDestroy보다 먼저 실행됨. OnNetworkSpawn에서 등록한 RPC 콜백 및 이벤트 구독 해제(-=) 등 메모리 해제에 사용함.

OnDisable() (MonoBehaviour): 오브젝트 비활성화 시 실행.

OnDestroy() (MonoBehaviour): 오브젝트 메모리 완전 해제 시 실행.

3. 실시간 변수 동기화 시스템 (NetworkVariable)
로컬 데이터 변경 사항을 타 클라이언트들에게 자동 동기화하기 위해 Netcode 제네릭 타입을 적용함.

선언 및 권한 제어
C#
NetworkVariable<float> playerAlpha = new NetworkVariable<float>( 0f, NetworkVariableReadPermission.Everyone, NetworkVariableWritePermission.Owner);
동기화 원리: NetworkObject 컴포넌트의 고유 ID를 매개로 서버가 변경 데이터를 수집 후 클라이언트에 브로드캐스팅함.

권한 설정: 읽기 권한 Everyone, 쓰기 권한 Owner 설정을 통해 클라이언트 로컬 컴퓨터에서의 직접적인 값 변경 권한을 부여함.

조작 및 구독: 내부 값 제어 시 .Value 프로퍼티를 거쳐야 함. 값 변경 시점의 동기화 연출을 위해 OnValueChanged 이벤트를 구독하여 애니메이터 파라미터나 UI 상태 갱신에 활용함.

4. VR 전용 UI 캔버스 및 레이캐스터 구성
World Space 캔버스 전환
Screen Space - Overlay 방식은 VR 환경에서 렌더링 오류를 발생시키므로, 모든 UI Canvas의 Render Mode를 World Space로 전환하여 3D 공간 내 배치함.



5. 레이캐스터 구성 분기
PC 테스트 환경: 화면 중앙 마우스 클릭 조작을 위해 기존 Graphic Raycaster를 임시 유지함.

VR 실기 환경: 컨트롤러 레이 포인터 입력을 수신하기 위해 XR Interaction Toolkit 패키지의 TrackedDeviceGraphicRaycaster 컴포넌트를 필수 배정함.

구성 단축: XR 탭의 UI Canvas 프리팹을 활용할 경우 두 컴포넌트가 자동으로 이중 매핑되어 빠른 세팅이 가능함.

6. VR 입력 및 인터랙션 시스템 (XR Interaction Toolkit)
🎮 Unity Input System & 컨트롤러 매핑
Input Action Asset 등록: VR 컨트롤러의 물리 입력을 시스템 이벤트와 연동하기 위해 Input Action Manager 컴포넌트에 필요한 Action Asset을 등록하고 정상 활성화함.

Select & Activate 입력 분리: 물체를 쥐고 유지하는 동작(Select)과, 손에 쥔 상태에서 트리거를 당겨 도구를 사용하는 동작(Activate)의 입력 처리를 명시적으로 분리함.

XR Device Simulator: 실제 VR 헤드셋 장비 없이 PC 환경에서 마우스와 키보드로 컨트롤러의 6DoF 회전/이동 및 버튼 입력을 테스트할 수 있도록 시뮬레이터 환경을 구축함.

🖐️ 물리 Grab 시스템 및 최적화
물리적 Jitter(떨림) 현상 해결: 가상 오브젝트를 쥐었을 때 발생하는 미세한 프레임 떨림을 방지하기 위해, XR Grab Interactable의 Movement Type을 Instantaneous 혹은 Kinematic으로 상황에 맞게 조정하고 Rigidbody 물리 설정을 제어함.

원격 잡기 (Force Grab): 원거리에 위치한 물체에 레이(Ray) 포인터를 조준하고 그립 버튼을 눌렀을 때, 오브젝트가 손으로 자연스럽게 끌어당겨져 쥐어지는 원격 조작 메커니즘을 설계함.

Attach Transform 기준점 조정: 물체를 잡았을 때 캐릭터 손의 앵커 포인트가 뒤틀리는 현상을 막기 위해, Grab 대상 오브젝트 내부에 빈 게임 오브젝트를 배치하고 Attach Transform 좌표 오프셋을 세밀하게 수동 세팅함.

🔌 소켓 인터랙터 및 입력 필터링
XRSocketInteractor 스냅(Snap): 열쇠, 도구 등 특정 인터랙티브 물체를 지정 공간에 가까이 가져갔을 때, 콜라이더 트리거 인식을 통해 자석처럼 제자리에 달라붙도록 고정하는 소켓 시스템을 구현함.

Interaction Layer Mask 적용: 소켓 영역에 관련 없는 엉뚱한 아이템이 잘못 장착되는 비정상 작동을 차단하기 위해, 물리 레이어와 별개로 XR 전용 Interaction Layer Masks를 설정하여 정밀 필터링을 완료함.

7. 네트워크 씬 로딩 시스템 (Netcode Scene Management)
🔄 Netcode Scene Loading 및 상태 관리
멀티플레이어 환경 내 클라이언트 간 동기화 및 안정적인 씬 전환을 위해 Unity Netcode 비동기 씬 로드 기능을 채택함. 반환 타입인 SceneEventProgressStatus (Enum)를 통해 로딩 파이프라인 상태를 추적하고 트러블슈팅을 진행함.

SceneEventProgressStatus.Started: 씬 전환 명령이 서버 및 클라이언트 네트워크 파이프라인에 정상적으로 전달되어 로딩이 시작됨(성공).

SceneEventProgressStatus.SceneEventInProgress: 이전 씬의 로드/언로드 연산이 완료되지 않은 상태에서 중복 씬 전환 요청 시 발생하는 예외 에러.

SceneEventProgressStatus.InvalidSceneName: 요청한 씬 이름의 철자 오류 또는 Build Settings의 씬 목록에 등록되지 않은 경우 발생하는 예외 에러.

🔄 Additive (중첩) 씬 로드 방식
⚠️ 중요: Additive 방식은 이전 씬을 자동으로 언로드해주지 않습니다.

기존 단일 씬 교체 방식의 프리징(화면 정지)으로 인한 VR 멀미를 차단하기 위해, 백그라운드에서 다음 씬 데이터를 안전하게 로드하는 LoadSceneMode.Additive 방식을 채택함.

새로운 로비 씬 로드가 끝나면, 연산 및 메모리 낭비를 방지하기 위해 기존의 부팅 씬을 코드로 직접 지우는 SceneManager.UnloadSceneAsync("Scene_Boot") 루틴을 수동으로 호출해야 함.

🔄 Active Scene의 수동 전환
비동기 씬 로드 완료 직후, 시스템이 새로 불려온 로비 씬을 메인 환경으로 완전 인지할 수 있도록 코드로 SceneManager.SetActiveScene()을 명시적 호출해주어야 조명(Lighting) 및 스카이박스 등 월드 환경 렌더링 세팅이 깨지지 않고 유지됨.
